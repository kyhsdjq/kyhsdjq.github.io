## 简介

RAG 即 Retrieval-Augmented Generation，旨在通过 **检索** 来增强 LLM 回答的准确性和可靠性，从而解决 LLM 的三个痛点：幻觉、知识过时、推理过程不可追溯。

RAG 大致分为三条技术路线：Pre-training、Inference、Fine-tuning，区别在于它们在不同阶段运用检索技术。
最早 Pre-training 随着 Transformer 出现，而目前最主流的技术路线是 Inference。随着研究进行，Fine-tuning 技术被尝试融入其中。
本文将聚焦于基于 Inference 的 RAG，并讨论它后续的优化方式。

## 基础架构

RAG 核心只有三步：indexing、retrieval、generation：
1. indexing：将知识分片，随后通过 embedding 模型生成向量，存入向量数据库
2. retrieval：从向量数据库中取出与 query 最接近的 k 片知识
3. generation：将知识插入 prompt 中，发送给 LLM 得到最终结果

以上架构常被称为 Naive RAG，实际应用中存在一些缺陷：

| 问题类型   | 具体描述                                   |
|:----------|:-------------------------------------------|
| 检索不准   | 选择无关段落而错失重要信息                    |
| 检索冗余   | 从多个来源检索到重复信息                      |
| 检索不足   | 面对复杂问题单次检索不够获取足够信息           |
| 上下文混乱  | 不同文档的风格和语气不同                     |
| 生成幻觉   | 生成内容与上下文无关                         |
| 生成浅薄   | 生成结果只是复述材料，缺少自己的洞察           |

## 实现细节

### 原始数据

| 数据类型         | 具体描述                                                                 |
|:--------------:|:-----------------------------------------------------------------------|
| 无结构数据       | 文本，来自百科全书、多语言文本、领域知识                                     |
| 半结构数据       | PDF，其中表格的分片以及语义提取都存在困难                                   |
| 结构数据         | 知识图谱，可以跳过 indexing 步骤直接检索                                   |
| LLM 生成数据    | 经过大量数据的训练，有时 LLM 生成的数据可能比文献中检索而来的质量更高               |

### 分片粒度

分片粒度非常影响 RAG 效果，切得太粗可能导致检索到矛盾的内容，而切得太细可能导致检索不到关键知识，抑或是将相关内容切开。

从粗到细，文本切分单位为：Document、Chunks、Proposition、Sentence、Phrase、Token；知识图谱切分单位为：sub-Graph、Triplet、Entity。 

## 优化迭代

### indexing

- Small2Big：index 时粒度较细，retrieve 时粒度较粗
- Metadata Attachments：添加文件元数据（页码、文件名、作者、时间），辅助过滤或是赋予权重
- Reverse HyDE：对文档先总结再 embed
- raptor: 对文档做多层汇总，从而对多文档相关问题也能很好地解答
- LightRAG: 利用 LLM 根据文档建图，再为其中每个点和边创建索引

### embedding

- Mix Retreival: 同时使用 sparse encoder 和 dense retriever。其中 encoder 对文档进行一些预处理（例如 BM25 将信息片段处理为词频向量），随后 retriever（BERT 架构的模型）再将它映射到最终的向量空间中。
- Fine-tuning Embedding Model：针对特殊领域的文档（如医疗、法律），先微调 embedding 模型以确保它能正确理解。
- Alignment：使用微调来对齐 retriever 和 generator，让 retriever 返回的结果不仅仅是“和问题最相关的”，还是“generator 最需要的”。但如果使用外部 API 或计算资源有限时，只能通过 adaptor 妥协（如 UPRISE、AAR、PRCA、BGM、PKG）。

### query

直接根据用户的 query 搜索效果显然不好，可以通过 LLM 来优化 query

- step-forward：拆分为复数个具体的 query。适用于复杂问题，将一个宽泛的问题分解为多个子问题，分别检索后综合答案。

    ```
    优化前：什么是机器学习？
    优化后：
    - 机器学习的定义是什么？
    - 机器学习有哪些主要类型？
    - 机器学习的应用场景有哪些？
    ```

- step-back：提炼为更抽象的 query。当原问题过于具体时，退一步思考更高层次的概念，帮助检索到更全面的背景知识。

    ```
    优化前：怎么做西红柿炒鸡蛋？
    优化后：家庭常见的烹饪方法有哪些？
    ```

- RAG Fusion：生成多条相似的 query，通过 RRF（Reciprocal Rank Fusion）算法综合分析查询的结果。多角度提问，提高召回率。

    ```
    优化前：如何提高模型准确率？
    优化后：
    - 如何提高模型准确率？
    - 提升模型性能的方法有哪些？
    - 模型优化的常见技巧
    - 改善预测精度的策略
    ```

- least-to-most：拆分为多个逐渐深入的问题，并且按顺序解决它们。从简单到复杂，前一个问题的答案作为后续问题的上下文。

    ```
    优化前：如何部署一个高可用的微服务架构？
    优化后：
    1. 微服务架构
    2. 微服务架构 - 微服务架构的核心组件
    3. 微服务架构 - 微服务架构的核心组件 - 微服务服务的高可用性
    4. 微服务架构 - 微服务架构的核心组件 - 微服务服务的高可用性 - 微服务部署的最佳实践
    ```

- HyDE：生成一篇虚假的文档，用它来搜索最近的知识。让 LLM 先生成一个假设的答案，用这个答案的向量去检索，往往比问题本身更接近真实文档。

    ```
    优化前：什么是 Transformer 的自注意力机制？
    优化后（生成假设答案）：
    Transformer 的自注意力机制是一种允许模型在处理序列时关注不同位置的方法。
    它通过 Query、Key、Value 三个矩阵计算注意力权重，使用点积和 softmax 
    来确定每个位置对当前位置的重要性。这种机制使模型能够捕捉长距离依赖关系...
    ```

### routing

选择不同路径
- logical routing: 让 LLM 选择要查询的数据库
- semantic routing: embed 之后选择要使用的 prompt（让 LLM 扮演不同的角色）

### query construction

数据库可以提供一些固定参数以供筛选，例如时间和长度。实现过程中可以将它们作为 function call 的参数来传入。

### context engineering

- reranking: 优先显示最重要的内容，排序方式包括硬指标（MRR）、BERT 系列模型、reranking 模型（Cohere）、通用 LLM
- context compression：通过小模型压缩 prompt；先通过小模型过滤弱相关内容，随后通过 LLM 按优先级重排

### 流程优化

- 迭代检索：基于原始问题和已生成的回答反复搜索。
- 递归检索：根据先前的搜索结果优化问题，从而逐步拆解问题。
- 自适应检索：LLM 自主决定是否搜索

## 讨论

### RAG vs Fine-tuning

两者比较如下：

||RAG|Fine-tuning|
|---|---|---|
|优点|支持动态更新知识、解释性强|深度定制模型风格|
|缺点|延迟高、伦理问题|需要大量计算资源、模型通用性能下降|

总体而言，针对无标签的数据集，RAG 效果总是好于 Fine-tuning。不过，如果将 RAG 和 Fine-tuning 互相结合，或许可以相互补充，从而得到最佳的性能。

### RAG vs Long Context

随着上下文窗口变大，有人质疑直接将文档放在上下文中效果也不差于使用 RAG。

这相当于通过注意力机制来检索关键信息，其在压测下表现一般，尤其是当问题依赖于多个分散的信息时。

此外，RAG 反应更快，消耗 token 更少，具备强可追溯性。

### 多模态 RAG

**引自参考文献 [1]**：

> **Image**. RA-CM3 stands as a pioneering multimodal model of both retrieving and generating text and images. BLIP-2 leverages frozen image encoders alongside LLMs for efficient visual language pre-training, enabling zero-shot image-to-text conversions. The "Visualize Before You Write" method employs image generation to steer the LLM's text generation, showing promise in open-ended text generation tasks.
>
> **Audio and Video**. The GSS method retrieves and stitches together audio clips to convert machine-translated data into speech-translated data. UEOP marks a significant advancement in end-to-end automatic speech recognition by incorporating external, offline strategies for voice-to-text conversion. Additionally, KNN-based attention fusion leverages audio embeddings and semantically related text embeddings to refine ASR, thereby accelerating domain adaptation. Vid2Seq augments language models with specialized temporal markers, facilitating the prediction of event boundaries and textual descriptions within a unified output sequence.
>
> **Code**. RBPS excels in small-scale learning tasks by retrieving code examples that align with developers' objectives through encoding and frequency analysis. This approach has demonstrated efficacy in tasks such as test assertion generation and program repair. For structured knowledge, the CoK method first extracts facts pertinent to the input query from a knowledge graph, then integrates these facts as hints within the input, enhancing performance in knowledge graph question-answering tasks.
